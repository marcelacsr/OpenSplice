6 About the C# Reference Guide {#mainpage_about}
7 ============================
8 
9 The C# Reference Guide provides details of the Vortex OpenSplice (Subscription
10 Paradigm for the Logical Interconnection of Concurrent Engines) Application
11 Programming Interfaces for the C# language.
12 
13 The C# Reference Guide describes the Data Centric Publish Subscribe (DCPS)
14 layer. The purpose of the DCPS is the distribution of data (publish/subscribe). The
15 structure of the DCPS is divided into five modules. Each module consists of several
16 classes, which in turn generally contain several operations.

33 How to use the C# API {#mainpage_howtouse}
34 =====================
35 
36 The API is currently compiled on the Windows platform using Microsoft's C# compiler. A port to Linux using
37 the Mono framework has also been used by the opensource community, but that is not yet part of the commercially
38 supported releases. However, it should be possible to run the Windows DLL on Mono, since Mono is binary compatible with Windows.
39 The resulting deliverable is a library file called dcpssacsAssembly.dll, which you can find in the
40 %OSPL_HOME%/bin directory alongside all the other Vortex OpenSplice product libraries.
41 
42 Since the DLL is a managed assembly, it is not found by Windows using the normal PATH environment variable.
43 In order for the .NET framework to find a library, this library must either be copied to the same directory
44 location as the executable that tries to load it, or the library must be copied into the Global Assembly Cache (GAC).
45 For details about the global assembly cache, and how to manage it, see the Wikipedia Global Assembly Cache information.
46 
47 The C# API is very similar to the other language bindings, with a couple of exceptions:
48 
49 * The C# API satisfies the C# coding conventions, meaning that:
50 - All interfaces start with the 'I' character.
51 - All operations are written in CamelBack notation ('CreateTopic' instead of 'create_topic')
52 - Class-Attributes are not implemented by explicit getter operations, but by C# properties.
53 * The C# API introduces operation overloading: many operations have multiple versions with different parameters:
54 - Choose the operation with the minimum amount of parameters that fits your need.
55 - Parameters that are not passed are set to their default values.
56 - This removes the need for some of the special purpose macro's that represent default values for parameters.
57 * Constants have been defined in the classes they refer to.
58 - For example: the DURATION_INFINITE is a constant called 'Infinite' in the struct Duration.
59 
60 To generate a C# data model from your Topic IDL definitions, use idlpp in the following way:
61 
62  idlpp -l cs -S < IDL_FILE >
63 
64 Here, 'cs' represents the Csharp language, and -S represents the fact that there is no dependency on any ORB. See also 'idlpp -h'.
65 
66 Current limitations of the C# API {#mainpage_currentlimitations}
67 =================================
68 
69 * idlpp is not able to generate the correct C# representation for IDL unions.
70 * Datatypes generated by idlpp are not always initialized to valid default values. For example,
71  an array of strings is initialized to a string array holding 'null' pointers, which you need
72  to assign manually before you can write the contents of your sample into the DDS.
